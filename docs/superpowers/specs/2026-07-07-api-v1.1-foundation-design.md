# External API v1.1 Foundation Design

## Status

Approved for implementation planning.

## Context

OpenAuto Companion already has the External API v1 protocol foundation:
vendored v1.0 protobuf inputs, protobuf-lite code generation, frame codec,
handshake helpers, TCP and WebSocket transports, an `ApiSessionClient`,
credential persistence helpers, and opt-in live TCP validation.

The deployed Prodigy head unit now includes the additive v1.1 fields and
proxy-route teardown behavior. Companion should update its local foundation so
future live pairing and service-report work can build on the real contract
without cutting the runtime service over from legacy `9876` in this slice.

## Goals

- Refresh Companion's vendored `app/src/main/proto/api/` copy from Prodigy
  `develop`, preserving Prodigy as the source of truth.
- Advertise External API minor version `1` in `ClientHello`.
- Send `TimeReport.timezone_id` when a timezone id is supplied.
- Persist the stable `ServerHello.server_id` for a vehicle when present.
- Persist `SystemStatus.display_width` and `display_height` for a vehicle when
  a system snapshot provides them.
- Keep runtime service traffic, theme transfer, pairing UI, and legacy `9876`
  behavior unchanged.

## Non-Goals

- No Companion service cutover to External API reports.
- No live pairing UI or automatic credential acquisition.
- No removal of legacy `9876`.
- No Prodigy repo edits from the Companion side.
- No new theme/wallpaper transfer path; that remains gated on the future
  head-unit web-config HTTP upload endpoint.

## Architecture

The slice stays in the existing `org.openauto.companion.net.api` foundation
package and data model. It updates the contract inputs, protocol builders, and
small persistence helper seams without changing foreground-service connection
selection.

`ApiHandshake` remains a pure state machine. It will request major `1`, minor
`1`, and continue to treat feature availability as field presence, not as a
minor-version comparison. The ready result already exposes the raw
`ServerHello`, so callers can read `server_id` without adding a second result
type.

`ApiReports.timeReport` gains an optional `timezoneId` parameter. The method
sets `timezone_id` only when the value is non-blank. Existing callers that only
pass `unixTimeMs` keep the same output shape, which preserves compatibility
with v1.0 servers and old tests.

`Vehicle` gains nullable `serverId` serialized as `server_id`. The existing
`id` remains the app's current local key so existing paired vehicles and UI
matching do not churn during this foundation slice. Future pairing/service
work can migrate the canonical vehicle key after it has a live `server_id` and
a user-safe duplicate/merge story.

`ApiPairingCredentialStore` remains the narrow persistence helper for API
ready results. It will persist `server_id` from `ServerHello` when present and
add a helper for applying `SystemStatus` display dimensions to a matched
vehicle. This keeps v1.1 state handling testable without introducing Android
service dependencies.

## Data Flow

1. Protos are refreshed from `../openauto-prodigy/proto/api/`.
2. Code generation exposes:
   - `ServerHello.hasServerId()` / `serverId`
   - `TimeReport.hasTimezoneId()` / `timezoneId`
   - `SystemStatus.hasDisplayWidth()` / `displayWidth`
   - `SystemStatus.hasDisplayHeight()` / `displayHeight`
3. Handshake sends `ClientHello.requested_api_version_minor = 1`.
4. Pairing or known-client connection returns `ServerHello`; persistence code
   stores `server_id` if present.
5. System-topic snapshots can be passed to the persistence helper; display
   dimensions update only when both width and height are present and positive.
6. Runtime legacy status push continues using `PiConnection` and
   `Protocol.buildStatus` until a later cutover plan.

## Error Handling

- Missing v1.1 fields are tolerated. Absence means "do not update that value."
- Blank `server_id` is ignored.
- Display dimensions are ignored unless both fields are present and positive.
- Existing terminal handshake handling for `AuthReject`, `Error`, and early
  close is preserved. The live invalid-auth terminal-frame blocker remains a
  head-unit validation concern and does not block normal pairing or known
  client validation work.

## Testing

Unit coverage will prove the slice without hardware:

- Vendored proto/codegen compile exposes the v1.1 accessors.
- `ApiHandshakeTest` asserts `ClientHello.requested_api_version_minor == 1`.
- `ApiReportsTest` covers timezone presence when supplied and absence when
  omitted or blank.
- `VehicleSerializationTest` covers `server_id` round trip and default null.
- `ApiPairingCredentialStoreTest` covers persisting `server_id` from
  `ServerHello` and applying `SystemStatus` display dimensions.
- Existing API transport/session tests remain unchanged except where helper
  fixtures need minor `1` or optional `server_id`.

Repository verification remains:

```bash
./gradlew :app:testDebugUnitTest :app:assembleDebug
```

The opt-in live instrumentation test can be tightened to expect
`api_version_minor >= 1` and `server_id` on a successful ready result once real
credentials are provided. Invalid known-client auth should still tolerate the
known terminal-frame delivery issue without failing unrelated foundation work.

## Implementation Boundaries

Files expected to change:

- `app/src/main/proto/api/*.proto`
- `app/src/main/proto/api/README.md`
- `app/src/main/java/org/openauto/companion/net/api/ApiHandshake.kt`
- `app/src/main/java/org/openauto/companion/net/api/ApiReports.kt`
- `app/src/main/java/org/openauto/companion/net/api/ApiPairingCredentialStore.kt`
- `app/src/main/java/org/openauto/companion/data/Vehicle.kt`
- focused tests under `app/src/test/java/org/openauto/companion/`
- opt-in live validation test if its assertions can remain credential-safe
- roadmap and session handoff docs after verification

Files intentionally not changed in this slice:

- `CompanionService.kt` runtime connection/report loop
- pairing screens and QR parser behavior
- `PiConnection`, `Protocol`, and `ThemeTransfer` legacy runtime behavior
- Prodigy repository source files

## Acceptance Criteria

- Local protos match the Prodigy v1.1 additive contract.
- Unit tests prove minor `1`, timezone report behavior, vehicle `server_id`
  serialization, and persistence of `server_id` plus display dimensions.
- Legacy runtime behavior remains available and untouched.
- `./gradlew :app:testDebugUnitTest :app:assembleDebug` passes.
- The session handoff records what changed, why, status, next steps, and
  verification results.
